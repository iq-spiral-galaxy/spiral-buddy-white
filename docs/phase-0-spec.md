# Phase 0 — Proof of Concept Spec

이 문서는 Phase 0가 정확히 무엇을 검증하려는지 정의한다. 기능을 추가할지 말지의 판단 기준이 여기에 있다.

## 가설 3개

Phase 0의 모든 코드는 이 셋 중 하나를 검증한다.

### H1. 로드맵 주도 세션이 그냥 채팅보다 학습 효과가 있다

**무엇이 다른지:** 일반 클로드 채팅은 사용자가 매번 "오늘 뭐 배울까"를 정해야 한다. spiral-buddy는 로드맵 + 진도 + 이전 노트를 보고 도구가 제안한다.

**무엇을 보면 검증인지:**
- 본인 사용 시 "오늘 뭐 공부하지" 망설이는 시간이 줄어드는가
- 로드맵 챕터 순서대로 진행하는 비율이 일관되게 유지되는가 (중구난방으로 안 빠지는가)
- 같은 양의 시간을 들였을 때 다룬 챕터 수가 더 많은가

### H2. 자동 생성된 노트가 나중에 다시 봤을 때 쓸 만하다

**무엇이 다른지:** 보통 학습 후 노트 정리는 가장 미루는 일이다. 도구가 만들어주면 미루지 않는다. 단, 만들어진 노트가 쓰레기면 의미 없다.

**무엇을 보면 검증인지:**
- 1주일 뒤 다시 열어봤을 때 무슨 얘기였는지 빠르게 재구성 가능한가
- "헷갈렸던 / 확인이 필요한 지점" 섹션이 다음 세션의 실제 시작점으로 작동하는가 (가장 중요)
- 직접 손으로 수정해야 하는 비율이 30% 미만인가

### H3. 나선형 감지가 의미 있는 연결을 찾는다

**무엇이 다른지:** "이전에 다룬 X 개념을 지금 학습할 Y와 연결하자"를 사람이 매번 기억하긴 어렵다.

**무엇을 보면 검증인지:**
- suggestion의 추천(같은 챕터 deeper / 다음 챕터 / cross-link)이 합리적이라고 느껴지는가
- cross-link 모드가 발동했을 때 "아 그렇네"라는 반응이 나오는가
- 일주일에 1번 이상 cross-link 추천이 정확히 들어맞는가

## 형태(form factor) 결정

처음엔 CLI로 검증을 시작했는데, 마크다운 렌더링/코드 하이라이팅/스크롤백 없는 환경에선 dogfooding이 안 됐다. **검증 도구가 사용 자체를 막으면 의미 없음**이라 로컬 웹앱으로 전환했다. 이게 Phase 0의 첫 번째 발견.

**현재 형태 (Phase 0.5):**
- Hono 서버 + 단일 HTML 페이지 + vanilla JS
- 마크다운/코드 하이라이팅: CDN으로 marked + highlight.js
- 빌드 파이프라인 없음 (브라우저가 ES 모듈 직접 로드, tsx가 서버 직접 실행)
- 한 사용자/한 머신 가정 (인증 없음, 멀티 유저 없음)

## 명시적으로 안 하는 것

Phase 0에서 의도적으로 빠진 것들. 가설 검증 전에 추가하면 안 됨.

- ❌ Electron / Tauri 패키징
- ❌ Obsidian 플러그인
- ❌ MCP 서버 (Phase 1)
- ❌ 음성 입력
- ❌ 이미지/PDF 학습 자료
- ❌ Spaced repetition / Anki export
- ❌ 사용자 인증 / 멀티 유저
- ❌ 학습 통계 대시보드
- ❌ 모바일 대응

## 의도적인 한계 (Phase 0.5)

- **세션 상태 인메모리** — 서버 재시작하면 active session 잃음. dogfooding 단계라 OK.
- **컨텍스트 윈도우 관리 없음** — 한 세션 30분 내외 가정. 길어지면 토큰 비용 선형 증가.
- **`/save` (중간 저장) 없음** — 한 번에 끝낸다고 가정.
- **노트 충돌 처리 약함** — 같은 날 같은 주제로 두 번 세션 → 파일명 충돌 가능.
- **에러 복구 약함** — 스트리밍 중간 API 실패 시 대화 잃을 수 있음.

## 검증 일정

본인 사용 기준:
- **1주차** — H1, H2 빠르게 확인. 안 되면 도구 자체 재고.
- **2-3주차** — H3 확인. 누적 노트 5-10개 쌓인 후 가능.
- **4주차** — 의사결정: Phase 1(MCP) 진행할지, 다른 방향으로 갈지, 폐기할지.
