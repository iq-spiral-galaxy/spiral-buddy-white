# 기여 가이드 (Contributing)

Spiral Buddy Blue에 관심 가져주셔서 감사합니다. 기여하기 전에 아래 내용을 꼭 확인해주세요.

## 📜 라이선스 & 기여 동의 (먼저 읽어주세요)

이 프로젝트는 현재 [MIT 라이선스](LICENSE)로 공개됩니다. **머지를 위해서는 [기여자 라이선스
동의서(CLA)](CLA.md)에 서명**해야 합니다(PR을 열면 CLA Assistant 봇이 자동 요청 — 아래
[CLA 서명 방법](#cla-서명-방법-cla-assistant-lite) 참고). 핵심만 요약하면:

- 귀하의 기여는 **공개 시점 기준 MIT로 공개**됩니다. 다만 [CLA](CLA.md) §1에 따라 프로젝트
  소유자는 기여를 **재라이선스(상용·독점·이중 라이선스 포함)** 할 권리를 보유합니다. 즉
  프로젝트는 향후 다른 라이선스로 전환될 수 있으나, **이미 특정 라이선스로 공개된 버전**은
  그대로 유지됩니다.
- 저작권은 귀하가 계속 **보유**합니다(양도 아님).
- 기여는 **무상**이며, **지분·보수·고용·동업 관계를 발생시키지 않습니다.**
- 본인이 작성했거나 제출할 적법한 권리가 있고, 제3자 권리를 침해하지 않음을 보증합니다.

> 동의할 수 없다면 PR을 제출하지 말아주세요. (이슈로 제안·논의는 언제든 환영합니다.)
> 회사를 대신해 기여한다면 [기업용 CLA](CORPORATE_CLA.md)도 확인하세요.

자세한 조항은 [CLA.md](CLA.md)에 있습니다.

### DCO와 CLA의 차이 (둘 다 필요)

| | DCO (sign-off) | CLA (서명 봇) |
|---|---|---|
| **무엇** | 커밋별 출처 자기 인증 | 재라이선스·특허·저작인격권·지분 조항 동의 |
| **방식** | 모든 커밋에 `Signed-off-by` | PR에서 **CLA Assistant 봇에 1회 서명** |
| **검사** | `git commit -s` + 자동 검사 | 봇이 서명 여부를 status check로 강제 |

> DCO만으로는 소유자가 재라이선스할 권리가 생기지 않습니다 — **CLA가 그 권리를 부여**합니다.
> 둘 다 충족해야 머지됩니다.

### CLA 서명 방법 (CLA Assistant Lite)

PR을 열면 **CLA Assistant 봇**이 자동으로 [CLA](CLA.md) 서명을 요청하는 코멘트를 답니다.
[CLA.md](CLA.md)를 읽으신 뒤, 봇이 안내한 다음 문구를 **그대로 이 PR에 코멘트**해주세요:

```
CLA 문서를 읽었으며 이에 동의하고 서명합니다
```

- 서명자·시각·동의한 CLA 버전이 이 저장소의 `cla-signatures` 브랜치(`signatures/v1/cla.json`)에
  기록됩니다. **외부 서비스 없이** 저장소 안에 보관됩니다.
- 한 번 서명하면 이후 PR은 자동 인식됩니다(매번 서명 X). 서명은 과거 기여에도 소급 적용됩니다.
- 회사를 대신해 기여한다면 [기업용 CLA](CORPORATE_CLA.md)도 함께 제출하세요.

## ✍️ DCO Sign-off (필수)

모든 커밋에는 [Developer Certificate of Origin](https://developercertificate.org/) 서명이
필요합니다. 커밋할 때 `-s` 플래그를 붙이면 자동으로 추가됩니다:

```bash
git commit -s -m "fix: ..."
```

그러면 커밋 메시지 끝에 다음 줄이 들어갑니다:

```
Signed-off-by: 이름 <you@example.com>
```

이미 만든 커밋에 빠졌다면:

```bash
git commit --amend -s          # 마지막 커밋
git rebase --signoff <base>    # 여러 커밋 일괄
```

PR의 모든 커밋에 sign-off가 있는지 **자동으로 검사**됩니다(누락 시 체크 실패).

> 참고: merge 커밋(부모 2개 이상)은 sign-off 검사에서 자동 면제됩니다. DCO·CLA 검사는
> **Pull Request에만** 적용되며(소유자의 main 직접 push는 의도적으로 우회), 외부 기여는
> 항상 PR로 제출해주세요.

## 🛠️ 개발 흐름

```bash
pnpm install
pnpm dev            # 웹앱 개발 모드
pnpm electron:dev   # Electron 개발 모드
pnpm build          # TypeScript 빌드 검증
```

- 코드 스타일은 주변 코드를 따라주세요(주석 밀도·네이밍·관용구).
- 큰 변경은 PR 전에 **이슈로 먼저 논의**해주세요.
- 클라이언트 vendor 라이브러리(marked/KaTeX 등) 버전을 올렸다면 `pnpm vendor`로
  `client/vendor/` 번들을 재생성하고 함께 커밋해주세요.

## 📦 제3자 코드 / 라이선스

외부 코드(오픈소스 스니펫 등)를 포함한다면:

- PR에 **출처와 라이선스를 명시**하고, 해당 라이선스가 프로젝트(MIT)와 **호환**되는지
  확인해주세요. 호환성 확인 책임은 기여자에게 있습니다.
- GPL 등 비호환 라이선스 코드는 받을 수 없으며, 발견 시 maintainer가 제거할 수 있습니다.
- 출처를 거짓으로 표시하면 [CLA](CLA.md) §4 보증 위반이 됩니다.

## ✅ PR 체크리스트

PR을 열면 템플릿의 체크박스가 뜹니다. 모두 확인해주세요:

- [ ] [CLA](CLA.md)에 서명 (PR을 열면 봇이 안내 — 위 [CLA 서명 방법](#cla-서명-방법-cla-assistant-lite))
- [ ] 모든 커밋에 DCO `Signed-off-by` 포함 (`git commit -s`)
- [ ] 본인이 작성했거나 제출할 적법한 권리 보유, 제3자 권리 비침해
- [ ] `pnpm build` 통과
